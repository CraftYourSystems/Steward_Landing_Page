# Homepage Specification

## 05 — One System. Every Part of Your Restaurant.

Version: 1.0

Status: Ready for Implementation

References

- Steward Constitution
- Homepage Architecture

---

# Purpose

This section explains how Steward connects the different parts of a restaurant into one continuous operational system.

Rather than presenting a list of features, it should illustrate how information naturally flows through the restaurant.

The visitor should leave understanding that Steward is one connected operating system—not a collection of independent tools.

---

# User Question

"How does Steward actually work?"

---

# Business Goal

- Explain Steward's ecosystem.
- Demonstrate product integration.
- Differentiate Steward from standalone restaurant software.
- Build understanding without overwhelming visitors.

---

# Core Message

Every part of a restaurant generates operational information.

Steward quietly connects those parts so that information flows naturally from customers to owners.

---

# Content

## Eyebrow

One Connected System

---

## Heading

One system.

## Supporting Copy

From the moment a customer scans a QR code until the day's final summary is prepared, Steward keeps every operational workflow connected through one dependable system.

---

# Layout

Centered section.

Large heading.

Supporting copy below.

Flow diagram positioned centrally.

Plenty of surrounding whitespace.

The diagram should be easy to understand within a few seconds.

---

# Visual Hierarchy

1. Operational Flow

2. Heading

3. Supporting Copy

4. Powered quietly by Needle

The flow diagram is the visual centerpiece.

---

# Visual Style

Minimal.

Editorial.

Simple connecting lines.

Clean icons.

Consistent spacing.

No decorative graphics.

No technical diagrams.

The illustration should feel approachable.

---

# Motion

Each step fades into view sequentially.

Connecting lines appear naturally.

No complex path animations.

No moving particles.

No continuous looping.

Motion should help users understand the flow.

---

# Interaction

Hovering over each stage may provide a brief explanation.

Example:

Kitchen Display

"Orders appear instantly for kitchen staff."

Interactions should remain optional.

The diagram should communicate effectively without interaction.

---

# Accessibility

Logical reading order.

Keyboard-accessible interactive elements.

Meaningful labels for screen readers.

The operational flow should remain understandable without animation.

---

# Copy Principles

Avoid technical terminology.

Use operational language restaurant owners already understand.

Describe outcomes rather than implementation.

Never explain APIs, databases, integrations, or infrastructure.

---

# Engineering Notes

Reusable flow component.

Responsive layout.

Vertical flow on mobile.

Horizontal flow on desktop where appropriate.

Smooth sequential animations using Framer Motion.

Future workflow additions should require minimal structural changes.

---

# Acceptance Criteria

Visitors should understand:

- Steward connects restaurant operations.
- Information flows naturally.
- Needle sits on top of the operational system.
- Steward is one platform rather than multiple disconnected tools.

The diagram should explain the ecosystem within a few seconds.

---

# Out of Scope

Architecture diagrams.

Technical infrastructure.

Backend services.

Database relationships.

API integrations.

System architecture.

Developer documentation.

These belong in technical documentation—not the homepage.

---

# Definition of Done

The section is complete when visitors naturally think:

"I don't need separate systems for ordering, kitchen operations, payments, and reporting.

Steward connects everything."
